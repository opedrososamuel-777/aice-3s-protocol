An advanced, enterprise-grade AI prompting framework and software engineering methodology designed to strip away LLM hallucinations and force generative models to behave as battle-tested Senior Software Architects, Rigorous QA Engineers, and Pessimistic Watchdogs.

## 🏛️ Architectural Hierarchy: The 3S Core

AICE-3S operates under an unyielding, non-negotiable structural hierarchy. If any user request or AI-generated implementation attempts to bypass a higher pillar for convenience, the protocol triggers an immediate system alert and forces structural refactoring.

1. 🔒 **SECURE (Security First)**
   * **Absolute Credential Isolation:** Explicit `.env` enforcement and restrictive `.gitignore` scaffolding from runtime second zero.
   * **Data Sanitization:** Strict runtime masking preventing PII, tokens, or hashes from ever leaking into stdout or debug traces.
   * **Preferred Stack:** Native prioritization of memory-safe, concurrent, compile-time hardened technologies (**Go/Golang**) and strictly audited, highly-validated Open Source libraries.

2. 🛡️ **STABLE (Resilience & SRE-Driven)**
   * **Preemptive FMEA Scaffolding:** Pre-code phase requires a strict *Failure Mode and Effects Analysis* aiming to catch and structural-kill $\ge 90\%$ of architectural edge-cases before execution.
   * **Hermetic Exception Guardrails:** Defensive programming pattern preventing runtime application panics/crashes. All caught faults are cleanly handled and delegated.
   * **Local Error Diagnostics Store:** Dedicated decoupled SQLite schema to persist unique error footprints, counters, and timestamps for localized, future historical debugging.
   * **Token-Optimized Markdown Logging:** Verbose application traces are parsed directly into lightweight Markdown formatting. This optimizes LLM contextual memory lookups and allows AI-driven observability tooling to parse logs with minimal token overhead.

3. 💡 **SIMPLE (Scalable Minimalist Design)**
   * **Zero AI Footprint:** Production code must be clean, modular, and idiomatic. Standard boilerplate and synthetic-looking multi-line comments generated by default AI routines are stripped. Documentation focus shifts to architectural *why* rather than syntax *what*.
   * **Cross-Platform Portability:** System design targeting strict agnostic runtime execution across heterogeneous nodes.

## 📦 Multi-Platform Deliverable Blueprint

For **EVERY** specified targeted operating system (e.g., Enterprise Linux distros, Windows, macOS) or deployment infrastructure (e.g., Docker, Kubernetes orchestration, bare-metal edge), the protocol mandates a comprehensive platform-specific documentation footprint containing:

*   **Platform-Specific FMEA Matrix:** Mapping host-level failure states (e.g., Linux storage permissions, Windows path delimiter conflicts, container memory limits) paired with concrete applied counter-measures.
*   **3-Tier Tri-Level Manual:**
    1.  *L1 - Non-Technical (Layman):* High-level functional concept and one-click execution playbooks.
    2.  *L2 - Intermediate Dev:* Local setup parameters, environment configurations (`.env`), basic operational CLI tooling, and standard troubleshooting pipelines.
    3.  *L3 - Senior/SRE:* Internal component directory layouts, physical SQLite error database mapping, concurrency runtime profiling, OS-specific memory tuning, and Markdown trace mapping.
*   **Unified Multi-Target CI/CD:** Native, cross-platform compilation, automated verification dry-runs, and reliable deployment integration workflows.
